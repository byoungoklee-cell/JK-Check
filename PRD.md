# PRD — 일한 번역검토 도구 (JP → KO Translation Review Tool)

## 1. 프로젝트 개요

### 1.1 목적
본 애플리케이션은 일본어 원문 Word 문서(`.docx`)와 이에 대응하는 한국어 번역문 Word 문서(`.docx`)를 업로드하여, 두 문서의 문장을 1:1로 자동 매칭하고 번역 품질(유사도)을 시각적으로 검토 및 수정할 수 있도록 지원하는 웹 기반 도구입니다.

### 1.2 주요 특징
*   **Pure Client-Side**: 서버 개입 없이 100% 브라우저 환경(HTML, CSS, JS)에서 동작.
*   **직관적인 UI/UX**: 순수 CSS Grid 기반의 안정적인 검토 테이블 구조 및 상세 유사도 시각화(막대그래프).
*   **원포인트 복원(DOCX Export)**: 수정한 한국어 번역 결과를 단순 텍스트가 아닌, 사용자가 처음에 업로드했던 원본 한국어 Word 문서의 서식을 그대로 유지한 채 `.docx` 형태로 다운로드.

---

## 2. 기술 스택 및 라이브러리

*   **Core**: HTML5, Vanilla JavaScript, CSS3
*   **문서 처리 라이브러리 (CDN 방식 활용)**:
    *   `mammoth.browser.min.js`: `.docx` 파일에서 원시 텍스트 및 HTML 요소를 추출하는 데 사용.
    *   `jszip.min.js`: `.docx` 파일 구조(ZIP)를 열어 내부 XML(`word/document.xml`)을 수정하고 다시 묶는 데 사용 (원본 서식 유지 다운로드 구현용).

---

## 3. 단계별 시스템 구조 및 모듈 (재개발 시 참고 가이드)

프로젝트는 명확한 역할 분담을 위해 여러 모듈(JS 파일)로 나뉘어 있습니다. 

### Step 1. 화면 및 레이아웃 정의 (`index.html`, `css/style.css`)
*   **레이아웃 구성**: 
    1.  **헤더 & 스텝 바**: 현재 진행 상태 표시 및 상단 액션 버튼 (다운로드, 새 파일).
    2.  **화면 1: 업로드 (`#screen-upload`)**: 드래그 앤 드롭 및 파일 탐색기 기반의 파일 업로드 기능 제공.
    3.  **화면 2: 처리 중 (`#screen-processing`)**: 분석/매칭 진행 상태를 텍스트 디버그 로그와 함께 실시간 표시.
    4.  **화면 3: 검토 (`#screen-review`)**: 4열(번호/상태, JP 원문, KO 번역문, 유사도) CSS Grid 형태의 리스트 영역 제공.
*   **디자인 시스템**: `#f3f4f6` 톤의 밝은 배경, 초록색 액센트 컬러(`var(--accent-green)`), `textarea` 클릭 시 자연스러운 포커스 효과 등 모던한 가독성 지향.

### Step 2. 텍스트 전처리 (`js/preprocessor.js`)
*   **문장 추출 (`extractSentences`)**: mammoth로 추출한 텍스트/HTML을 줄바꿈 단위로 분리.
*   **정규화 (`normChar`)**: 전각 기호 및 알파벳/숫자를 반각으로 통일시켜 유사도 비교 성능 최적화.
*   **태그 제거/식별 (`stripTags`, `isTagLine`)**: `【０００１】`, `〔１〕` 등 특허/기술 문서에 흔히 등장하는 단락 번호 태그를 식별하고 비순수 본문 줄을 필터링. (이때 식별된 태그 라인 정보는 추후 DOCX 조립 시 무시할 수 있도록 외부 모듈로 내보냄)

### Step 3. 유사도 계산 (`js/similarity.js`)
*   단순 문자열 비교가 아닌 구조적 유사도를 측정하는 알고리즘 구현.
*   **가중치 지표**: 길이(25%), 숫자(30%), 알파벳(20%), 기호(10%), 문서 내 위치(15%).
    *   *길이 최적 비율 동적 적용*: 일본어 원문의 길이가 20자 이하인 단문은 1.0(1:1)을 최적 비율로 간주하며, 20자를 초과하는 일반 문장은 번역 시 길어지는 특성을 반영하여 1.2배를 최적 비율로 평가.
*   **LCS (Longest Common Subsequence) 활용**: 숫자, 영단어, 특수기호(`<>` 문자 포함)를 모아 각각의 시퀀스에 대해 LCS 알고리즘을 적용, 순서와 집합 일치도를 동시에 평가.

### Step 4. 문장 자동 매칭 (`js/matcher.js`)
*   일본어 원문 배열과 한국어 번역문 배열을 받아 최적의 1:1 매칭 쌍 탐색.
*   **Dynamic Programming (DP)** 기반 매칭: 
    *   두 문장이 매칭될 때 `similarity.calculate`에 따른 유사도를 점수에 합산.
    *   한쪽 문장을 건너뛸 경우 패널티(`-0.1`)를 부여.
    *   최종적으로 누적 점수를 최대화하는 경로를 역추적하여 `{ jp, kr, 타입, 인덱스 }` 형태의 매칭 객체 배열 생성. 미매칭인 경우 빈 문자열 할당.

### Step 5. 메인 컨트롤러 (`js/app.js`)
*   **전역 상태 관리**: 업로드된 `File` 객체, 매칭된 문장 쌍 데이터(`pairs`), 점수 배열(`simScores`), 각 줄의 상태 및 피드백 텍스트(`krTexts`).
*   **화면 전환 흐름 (Flow)**:
    1.  양쪽 문서(JP/KO)가 업로드되면 "검토 시작" 활성화.
    2.  로딩 진입 시 텍스트 추출 → 전처리 → 매칭 → 유사도 계산 일괄 수행 (로깅).
    3.  0.3초 딜레이 후 검토 화면(`Reviewer.init()`)으로 자동 전환.
*   **외부 API 파이프라인**: UI 컴포넌트 간 데이터 연동을 위해 `getKrFile`, `setRowKrText` 등의 헬퍼 함수 제공.

### Step 6. 리뷰어 UI 로직 및 데이터 처리 (`js/reviewer.js`)
*   **동적 DOM 렌더링**: 검토 리스트 영역에 `innerHTML` 스트링 조작 방식으로 수백~수천 개의 행을 한 번에 효과적으로 렌더링. 이벤트 핸들러는 최상위 부모에 위임(Event Delegation)하여 성능 확보.
*   **편집 및 상태 관리 로직**:
    *   배지(미검토, 정상, 주의, 오류) 클릭을 통한 상태 부여 기능.
    *   한국어 텍스트 패널 클릭 시 해당 div를 자동 리사이즈 지원 `textarea`로 스왑 (인라인 에디팅 기능).
*   **Word 문서(`docx`) 생성 엔진 (`downloadTranslation`)**:
    1.  `JSZip`을 이용해 사용자가 최초 업로드한 한국어 `.docx` 버퍼를 읽기.
    2.  내부의 `word/document.xml` 요소를 DOMParser로 분석해 단락(`<w:p>`) 목록 순회.
    3.  `mammoth`가 텍스트를 무시한 패턴이나 `isTagLine`에 해당하는 줄은 패스하여 추출 인덱스와 XML 노드의 인덱스 동기화.
    4.  매칭된 단락에서 텍스트 노드(`<w:t>`)를 싹 비우고, `【０００１】` 같은 식별 태그는 보존한 채로 편집된 최종 한국어 번역 문자열 삽입.
    5.  XMLSerializer로 재조장 후 JSZip으로 `.docx` Blob 생성 및 다운로드. (원본 레이아웃, 테이블, 폰트 100% 유지 달성).

---

## 4. 향후 유지보수 시 주의사항 (Troubleshooting Points)

1.  **`<w:t>` 파편화 (DOCX XML)**: 
    Word 문서는 문장 하나를 여러 `<w:t>` 태그로 분할 저장하는 패턴을 가집니다. 이를 `reviewer.js`의 `downloadTranslation` 에서 첫 번째 `<w:t>`에 텍스트를 몰아놓고 나머지 `<w:t>`를 삭제하는 방식으로 해결했습니다. 로직 수정 시 주의가 필요합니다.
2.  **`isTagLine` 외부 참조 여부**: 
    다운로드 시 XML 매칭 인덱스(`currentKoIdx`)의 싱크가 틀어지지 않으려면 전처리 모듈(`preprocessor.js`)에서 걸러지는 기준(TagLine 무시 로직)과 완벽하게 동일한 룰을 타야 합니다. 따라서 `window.Preprocessor.isTagLine`의 노출 및 의존성 유지가 중요합니다.
3.  **HTML 태그 기호 특수문자 파싱 버그 방지**:
    `similarity.js`의 `SYM_SET` 배열 정의 시 `<` 와 `>` 가 `&lt;` 및 `&gt;` 와 같은 HTML 엔티티로 들어갈 경우 LCS 유사도 검출이 무너집니다. 반드시 실제 문자로 선언해야 합니다.
